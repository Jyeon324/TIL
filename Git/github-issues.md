# [Git] GitHub Issues

## 1. GitHub Issues란 무엇인가?

-   **GitHub Issues**는 GitHub 저장소(Repository) 내에 존재하는 프로젝트 관리 및 협업을 위한 도구이다.
-   개발자, 기획자, 디자이너 등 프로젝트에 참여하는 모든 사람이 이곳에서 아이디어를 제안하고, 문제를 공유하며, 작업 진행 상황을 투명하게 관리할 수 있다.
* 예시는 다음과 같다
  *  `버그 리포트`: "로그인 페이지에서 이메일 유효성 검사가 작동하지 않음"
  *  `새로운 기능 제안`: "소셜 로그인 기능 추가"
  *  `문서화 작업`: "API 엔드포인트 A, B, C에 대한 문서 작성"
  * ️`리팩토링`: "기존 AuthService 코드를 CQRS 패턴으로 개선"
  *  `질문`: "A 모듈의 의존성 주입 방식에 대한 질문"
  *  `배포 작업`: "v1.1.0 릴리즈 준비"
-   `Issues`는 프로젝트의 '할 일 목록(To-do list)'이자, '공개 토론장'이며, '진행 상황 추적기'의 역할을 모두 수행하는 **중앙 커뮤니케이션 허브**이다.

---

## 2. 왜 GitHub Issues를 사용해야 하는가?

프로젝트를 진행할 때 카카오톡, 슬랙, 이메일, 노션 등 여러 채널로 소통하면 논의가 흩어지고 히스토리 추적이 어려워진다. GitHub Issues는 이러한 문제점을 해결해준다.

1. **논의의 중앙화 (Centralization):**
   * 모든 작업 요청과 토론이 코드가 저장된 **저장소(Repository)와 동일한 공간**에서 이루어진다.
   * 개발자는 코드를 보다가 관련된 이슈를, 이슈를 보다가 관련된 코드를 즉시 찾아볼 수 있다.
2. **투명한 협업 (Transparency):**
   * 팀원 누구나 현재 어떤 작업이 진행 중인지, 누가 담당하고 있는지, 어떤 논의가 오갔는지 확인할 수 있다.
   * 오픈소스 프로젝트의 경우, 외부 기여자들도 프로젝트의 방향성을 이해하고 참여하기 쉬워진다.
3. **체계적인 작업 관리 (Systematic Management):**
   * 이슈에 담당자(Assignee), 라벨(Label), 마일스톤(Milestone) 등을 할당하여 작업을 체계적으로 분류하고 우선순위를 정할 수 있다.
4. **코드와의 강력한 연결성 (Traceability):**
   * Issues의 가장 강력한 기닁이다. 
   * **"이 코드는 왜 바뀌었는가?"**라는 질문에 대한 답을 제공한다.
   * 특정 커밋(Comiit)이나 풀 리퀘스트(Pull Request)가 어떤 이슈를 해결하기 위해 작성되었는지 명확하게 연결하고 추적할 수 있다.

---


1.  **가독성 (History):** 커밋 목록(`git log`)만 봐도 어떤 변경사항이 있었는지 한눈에 파악할 수 있다.
2.  **검색 용이:** 특정 유형(e.g., `fix` 또는 `feat`)의 커밋만 필터링해서 찾아보기 쉽다.
3.  **협업:** 팀원 모두가 동일한 규칙으로 메시지를 작성해 소통 비용이 줄어든다.
4.  **자동화:** 커밋 메시지를 기반으로 **버전 릴리스 노트**나 **CHANGELOG**를 자동으로 생성할 수 있다. (매우 강력한 기능)

## 3. 기본 작성법 및 핵심 구성 요소

GitHub 저장소의  **[Issues]** 탭에서 **[New issue]** 버튼을 눌러서 새 이슈를 생성할 수 있다. 좋은 이슈는 '제목과 '본문', 그리고 '사이드바'의 3요소로 구성된다.

### 3.1. 제목 (Title)

* 이슈의 핵심 내용을 **명확하고 간결하게** 요약해야 한다.
* (나쁜 예: "작동 안 됨")
* (좋은 예: "[Bug] /cart 페이지에서 수량 변경 시 가격이 업데이트되지 않음")
* (Tip) `[Bug]`, `[Feature]` 같은 말머리(Prefix)를 사용하면 이슈 목록을 한눈에 파악하기 좋다.

### 3.2. 본문 (Body)

* 이슈의 상세 내용을 **마크다운(Markdown)**으로 작성한다.
* **버그 리포트**라면 '재현 단계', '기대 결과', '실제 결과', '환경(OS, 브라우저)' 등을 포함해야 한다.
* **기능 제안**이라면 '제안 배경(Why)', '기능 상세 설명(What)', '참고 자료(As-is, To-be)' 등을 포함해야 한다.
* (Tip) 이슈 템플릿을 사용하면 이 과정을 자동화할 수 있다. (심화 섹션에서 설명)

### 3.3. 사이드바 (Sidebar) - 이슈의 메타데이터

이슈를 생성할 때 우측에 보이는 사이드바는 이슈를 관리하기 위한 핵심 기능들의 집합이다.

* **Assignees (담당자):** 이 이슈를 처리할 사람을 지정한다. (최대 10명)
* **Labels (레이블):** 이슈를 분류하는 '태그'이다. (예: `bug`, `feature`, `priority:high`)
* **Projects (프로젝트):** 이슈를 칸반 보드 등에 연결하여 시각적으로 관리한다.
* **Milestone (마일스톤):** 여러 이슈를 묶는 상위 목표(예: 'v1.0 릴리즈')이다.
* **Linked pull requests (연결된 PR):** 이 이슈와 관련된 코드 변경 사항(PR)을 연결한다.

## 4. 예시: 좋은 이슈 vs 나쁜 이슈


### ❌ 나쁜 이슈 (Bad Issue)

**제목:** 프로필이 이상함
**본문:**
마이페이지 갔는데 프로필 사진이 안 떠요.
어제는 됐는데 왜 이러죠? 빨리 고쳐주세요.

> **(문제점)**
> * **맥락 없음:** 어떤 환경(PC/모바일?)인지, 어떤 사용자 계정인지 알 수 없습니다.
> * **추측 불가:** '안 뜬다'는 것이 X박스로 뜨는 것인지, 로딩만 도는 것인지 알 수 없습니다.
> * **비협조적:** 개발자가 문제를 재현하기 위해 다시 질문해야 하므로 시간 낭비가 발생합니다.

<br>

### ✅ 좋은 이슈 (Good Issue) - 버그 리포트

**제목:** [Bug] /mypage 에서 JPG 외 이미지 업로드 시 프로필 사진이 깨져 보임
**본문:**

### 1. 문제 요약
PNG, GIF 등 JPG가 아닌 형식의 이미지를 프로필 사진으로 업로드하면, 업로드는 성공하나 마이페이지에서 이미지가 깨진 아이콘(X박스)으로 표시됩니다.

### 2. 재현 단계
1.  테스트 계정(ID: `tester@example.com`)으로 로그인합니다.
2.  /mypage 로 이동하여 '프로필 사진 변경'을 클릭합니다.
3.  `test.png` (PNG 파일)을 업로드합니다.
4.  업로드 완료 후 마이페이지를 새로고침합니다.

### 3. 기대 결과
업로드한 `test.png` 이미지가 프로필 사진 영역에 정상적으로 보여야 합니다.

### 4. 실제 결과
프로필 사진 영역에 이미지가 로드되지 않고 깨진 아이콘으로 표시됩니다. (콘솔 확인 시 404 Not Found 발생)

### 5. 환경
* **OS:** Windows 11
* **Browser:** Chrome 120.0.1
* **Test Server:** dev
* **Screenshot:** `[screenshot.png]`

## 5. 심화 기능

프로젝트를 '회사 수준'에서 관리하기 위해 반드시 알아야 할 심화 기능이다. (이 부분은 GPT의 도움을 받아 작성해서 정확하지 않음.)

### 5.1. 전략적인 레이블 (Labels) 활용

레이블은 이슈를 **필터링**하고 **우선순위**를 정하는 가장 중요한 도구이다. GitHub는 기본 레이블(bug, documentation 등)을 제공하지만, 프로젝트에 맞게 커스텀하는 것이 좋다.

**추천 레이블 시스템 (Prefix 활용):**

* **Type (유형):** 이 이슈가 무엇에 대한 것인가?
    * `Type: Bug 🐛`
    * `Type: Feature ✨`
    * `Type: Docs 📚`
    * `Type: Refactor ♻️`
    * `Type: Chore 🧹` (빌드, 설정 등 기타 작업)
* **Priority (우선순위):** 얼마나 급한가?
    * `Priority: Critical 🚨` (즉시 수정)
    * `Priority: High ⬆️`
    * `Priority: Medium ↔️`
    * `Priority: Low ⬇️`
* **Status (상태):** 지금 어떤 상태인가?
    * `Status: In Progress 🏃` (작업 중)
    * `Status: Needs Review 🔍` (리뷰 필요)
    * `Status: Blocked 🚧` (다른 작업으로 인해 보류)
* **(Tip) `Good First Issue`:** 오픈소스에서 신규 기여자가 참여하기 좋은 쉬운 이슈에 이 레이블을 붙여준다.

### 5.2. 목표 관리 (Milestones)

마일스톤은 '**특정 날짜까지 완료해야 할 이슈들의 묶음**'이다.

* **활용 예:** "v1.0 정식 릴리즈", "2차 스프린트 (11/4~11/17)", "MVP 기능 구현"
* **핵심 기능 (진행률 표시):** 마일스톤에 10개의 이슈를 할당하고 5개의 이슈를 `Closed`하면, 마일스톤 페이지에 **진행률 50%**가 그래프로 표시된다.
* 이는 팀 리더나 PM이 프로젝트의 전체 진행 상황을 한눈에 파악하는 데 매우 유용하다.

### 5.3. 효율적인 코멘트 (Comments) 활용

코멘트는 단순한 댓글이 아닌, **'작업의 히스토리'** 그 자체이다.

* **@Mention (멘션):** `@username`을 입력하여 특정 팀원에게 알림을 보낸다. (예: `@backend-dev 이 부분 API 명세 확인 부탁드립니다.`)
* **Quote Reply (인용 답글):** 상대방의 긴 코멘트 중 특정 부분만 드래그한 후 `R` 키를 누르면, 해당 부분만 인용하여 답글을 달 수 있다.
* **Reactions (반응):** 👍, ❤️, 🎉
    * (Tip) "확인했습니다", "감사합니다" 같은 단순 알림은 코멘트로 남기기보다, 해당 코멘트에 👍 반응을 눌러 '읽었음'을 표시하는 것이 좋다. (알림 노이즈 감소)

### 5.4. 코드와 이슈 연결 (Linked Pull Requests)

이슈는 '문제 정의(What)', PR은 '코드 해결(How)'입니다. 이 둘을 연결하는 것은 협업의 핵심이다.

* **매직 키워드 (Magic Keywords):**
    * PR의 본문이나 커밋 메시지에 `Closes #이슈번호`, `Fixes #이슈번호`, `Resolves #이슈번호`를 작성한다.
    * **예:** `Closes #42`
    * **효과:** 이 PR이 `main` 브랜치에 **머지(Merge)되는 순간, 연결된 42번 이슈가 자동으로 `Closed` 상태가 된다.**
* **추적성:**
    * **(이슈에서 → PR로):** 42번 이슈 페이지만 봐도, "이 이슈를 해결하기 위해 어떤 PR이 진행 중인지" 링크가 표시된다.
    * **(PR에서 → 이슈로):** PR을 리뷰할 때, "이 코드를 왜 이렇게 바꿨지?"라는 맥락(Context)을 파악하기 위해 연결된 42번 이슈를 클릭해서 확인할 수 있다.

### 5.5. 일관성 강제 (Issue Templates)

매번 "좋은 이슈"를 작성하도록 교육하는 것보다, **템플릿을 만들어 강제하는 것**이 훨씬 효율적이다.

* **설정:** 저장소 루트에 `.github/ISSUE_TEMPLATE/` 폴더를 만들고, 그 안에 `bug_report.md`, `feature_request.md` 같은 마크다운 파일을 만든다.
* **작동:** [New Issue] 버튼을 누르면, 어떤 템플릿(버그 리포트? 기능 제안?)으로 작성할지 선택지가 뜬다.
* **예시 (`bug_report.md`):**
    ```yaml
    ---
    name: 🐛 버그 리포트
    about: 버그 발생 시 이 템플릿을 사용해 주세요.
    title: "[Bug] "
    labels: "Type: Bug 🐛, Priority: Medium ↔️"
    ---

    ### 1. 문제 요약
    (문제를 한 문장으로 요약해 주세요)

    ### 2. 재현 단계
    1. ...
    2. ...

    ### 3. 기대 결과
    (정상 동작 시 결과를 설명해 주세요)

    ### 4. 실제 결과
    (현재 발생하는 문제를 설명해 주세요)
    ```
    * (Tip) 최근에는 `YAML` 형식을 사용한 **Issue Forms**를 통해 텍스트 입력, 드롭다운, 체크박스 등을 만들 수 있어 훨씬 강력하다.

### 5.6. 이슈 vs. 디스커션 (Issues vs. Discussions)

* **Issues:** **실행 가능한 작업(Actionable tasks)**이다. (예: 버그 수정, 기능 개발)
* **Discussions:** **자유로운 토론(Open-ended conversations)**이다. (예: "우리 프로젝트에 타입스크립트를 도입하는 게 어떨까요?", "Q&A 게시판")
* **흐름:** Discussions에서 논의가 충분히 이루어져 실제 '작업'이 필요하다고 결정되면, 해당 토론을 기반으로 새 이슈를 생성(Create issue from discussion)할 수 있다.

## 6. 회고 (Retrospective)

### 6.1. GitHub Issues를 배우며 느낀 점

* 이슈가 단순히 버그 리포팅 툴이 아니라, 프로젝트 관리의 핵심이자 '중앙 허브'라는 것을 알았다.
* '좋은 이슈'를 작성하는 것이 왜 중요한지 알게 되었다. 불명확한 이슈는 개발자 간의 불필요한 커뮤니케이션 비용을 몇 배로 증가시킨다.
* Labels, Milestones, Linked PRs 기능은 단순히 '있으면 좋은' 기능이 아니라, 프로젝트의 성숙도를 결정하는 '필수' 기능임을 느꼈다.

### 6.2. 앞으로 나의 프로젝트에 적용할 것

1. **전략적 Label 시스템 구축:** `Type`, `Priority`, `Status` 3가지 축으로 레이블을 재정비하여 이슈 필터링을 습관화하겠다.
2. **PR/Commit 메시지 습관화:** 모든 커밋과 PR에 관련 이슈 번호를 연결(`Ref #10`)하고, PR이 이슈를 해결한다면 `Closes #10` 키워드를 사용하겠다.
3. **Issue-Driven Development:** 앞으로는 어떤 코드를 작성하기 전에, 먼저 이슈를 생성하고 그 이슈를 바탕으로 브랜치를 파서 작업하는 '이슈 중심 개발' 습관을 들이겠다.
