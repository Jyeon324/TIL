# [Git] GitHub Issues

## 1. GitHub Issues란 무엇인가?

-   **GitHub Issues**는 GitHub 저장소(Repository) 내에 존재하는 프로젝트 관리 및 협업을 위한 도구이다.
-   개발자, 기획자, 디자이너 등 프로젝트에 참여하는 모든 사람이 이곳에서 아이디어를 제안하고, 문제를 공유하며, 작업 진행 상황을 투명하게 관리할 수 있다.
* 예시는 다음과 같다
  *  `버그 리포트`: "로그인 페이지에서 이메일 유효성 검사가 작동하지 않음"
  *  `새로운 기능 제안`: "소셜 로그인 기능 추가"
  *  `문서화 작업`: "API 엔드포인트 A, B, C에 대한 문서 작성"
  * ️`리팩토링`: "기존 AuthService 코드를 CQRS 패턴으로 개선"
  *  `질문`: "A 모듈의 의존성 주입 방식에 대한 질문"
  *  `배포 작업`: "v1.1.0 릴리즈 준비"
-   `Issues`는 프로젝트의 '할 일 목록(To-do list)'이자, '공개 토론장'이며, '진행 상황 추적기'의 역할을 모두 수행하는 **중앙 커뮤니케이션 허브**이다.

---

## 2. 왜 GitHub Issues를 사용해야 하는가?

프로젝트를 진행할 때 카카오톡, 슬랙, 이메일, 노션 등 여러 채널로 소통하면 논의가 흩어지고 히스토리 추적이 어려워진다. GitHub Issues는 이러한 문제점을 해결해준다.

1. **논의의 중앙화 (Centralization):**
   * 모든 작업 요청과 토론이 코드가 저장된 **저장소(Repository)와 동일한 공간**에서 이루어진다.
   * 개발자는 코드를 보다가 관련된 이슈를, 이슈를 보다가 관련된 코드를 즉시 찾아볼 수 있다.
2. **투명한 협업 (Transparency):**
   * 팀원 누구나 현재 어떤 작업이 진행 중인지, 누가 담당하고 있는지, 어떤 논의가 오갔는지 확인할 수 있다.
   * 오픈소스 프로젝트의 경우, 외부 기여자들도 프로젝트의 방향성을 이해하고 참여하기 쉬워진다.
3. **체계적인 작업 관리 (Systematic Management):**
   * 이슈에 담당자(Assignee), 라벨(Label), 마일스톤(Milestone) 등을 할당하여 작업을 체계적으로 분류하고 우선순위를 정할 수 있다.
4. **코드와의 강력한 연결성 (Traceability):**
   * Issues의 가장 강력한 기닁이다. 
   * **"이 코드는 왜 바뀌었는가?"**라는 질문에 대한 답을 제공한다.
   * 특정 커밋(Comiit)이나 풀 리퀘스트(Pull Request)가 어떤 이슈를 해결하기 위해 작성되었는지 명확하게 연결하고 추적할 수 있다.

---


1.  **가독성 (History):** 커밋 목록(`git log`)만 봐도 어떤 변경사항이 있었는지 한눈에 파악할 수 있다.
2.  **검색 용이:** 특정 유형(e.g., `fix` 또는 `feat`)의 커밋만 필터링해서 찾아보기 쉽다.
3.  **협업:** 팀원 모두가 동일한 규칙으로 메시지를 작성해 소통 비용이 줄어든다.
4.  **자동화:** 커밋 메시지를 기반으로 **버전 릴리스 노트**나 **CHANGELOG**를 자동으로 생성할 수 있다. (매우 강력한 기능)

## 3. 기본 작성법 및 핵심 구성 요소

GitHub 저장소의  **[Issues]** 탭에서 **[New issue]** 버튼을 눌러서 새 이슈를 생성할 수 있다. 좋은 이슈는 '제목과 '본문', 그리고 '사이드바'의 3요소로 구성된다.

### 3.1. 제목 (Title)

* 이슈의 핵심 내용을 **명확하고 간결하게** 요약해야 한다.
* (나쁜 예: "작동 안 됨")
* (좋은 예: "[Bug] /cart 페이지에서 수량 변경 시 가격이 업데이트되지 않음")
* (Tip) `[Bug]`, `[Feature]` 같은 말머리(Prefix)를 사용하면 이슈 목록을 한눈에 파악하기 좋다.

### 3.2. 본문 (Body)

* 이슈의 상세 내용을 **마크다운(Markdown)**으로 작성한다.
* **버그 리포트**라면 '재현 단계', '기대 결과', '실제 결과', '환경(OS, 브라우저)' 등을 포함해야 한다.
* **기능 제안**이라면 '제안 배경(Why)', '기능 상세 설명(What)', '참고 자료(As-is, To-be)' 등을 포함해야 한다.
* (Tip) 이슈 템플릿을 사용하면 이 과정을 자동화할 수 있다. (심화 섹션에서 설명)

### 3.3. 사이드바 (Sidebar) - 이슈의 메타데이터

이슈를 생성할 때 우측에 보이는 사이드바는 이슈를 관리하기 위한 핵심 기능들의 집합이다.

* **Assignees (담당자):** 이 이슈를 처리할 사람을 지정한다. (최대 10명)
* **Labels (레이블):** 이슈를 분류하는 '태그'이다. (예: `bug`, `feature`, `priority:high`)
* **Projects (프로젝트):** 이슈를 칸반 보드 등에 연결하여 시각적으로 관리한다.
* **Milestone (마일스톤):** 여러 이슈를 묶는 상위 목표(예: 'v1.0 릴리즈')이다.
* **Linked pull requests (연결된 PR):** 이 이슈와 관련된 코드 변경 사항(PR)을 연결한다.
